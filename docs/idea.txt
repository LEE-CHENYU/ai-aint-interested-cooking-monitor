The hardest challenges (and how to de-risk in 8 hours)

1) Vision reliability in a real kitchen

Why it’s hard: glare, steam, occlusion by hands, variable lighting, clutter.

De-risk tactics:
	•	Zone cameras: treat each viewpoint as a “zone” (stove / counter / sink) and only detect 1–2 things per zone.
	•	Low FPS + smoothing: 2–5 fps per stream plus temporal smoothing beats flaky per-frame predictions.
	•	Event-driven: use cheap motion/scene-change detection to decide when to run heavier inference.

2) “Multitask coordination” is a planning problem, not just perception

If you try to solve general cooking scheduling, you’ll run out of time.

De-risk tactics:
	•	Start with one structured scenario: two timers + one interrupt.
	•	Use a simple heuristic scheduler (earliest deadline / slack time) and let the LLM handle phrasing + minor adjustments.

3) False alarms kill trust

If it nags constantly, people disable it.

De-risk tactics:
	•	Add a cooldown per alert type.
	•	Require persistence (e.g., “boiling” for N consecutive frames).
	•	Use severity levels: only interrupt with voice for high severity; show low severity on screen.

4) Demonstrating “fine-tuned” credibly

Judges will ask: “What exactly did you fine-tune?”

Two realistic options:
	•	Option A (recommended): fine-tune the agent policy (FunctionGemma / Gemma 3n)
Inputs: structured state + timers + recipe step context
Output: tool calls (set timer / speak / notify / reorder step)
This is compact, controllable, and clearly “adapted to the task.”
	•	Option B: fine-tune a tiny vision classifier (boiling vs not, smoke vs none)
Easier compute-wise, but it looks less “agentic” unless paired with strong action logic.

5) On-device compute constraints

Multi-view video + LLM can be heavy.

De-risk tactics:
	•	Run the LLM only on state changes (or every 2–5 seconds), not every frame.
	•	Quantize the LLM (4-bit) and keep prompts short.
	•	Keep the perception model lightweight and ROI-based.

A concrete system design that matches the brief

1) Perception layer (per camera/zone)

Each camera produces a compact state object (don’t pass raw pixels into the planner unless you have a multimodal Gemma working reliably).

Example state schema:

{
  "zone": "stove",
  "ts": 1730000000,
  "signals": {
    "pot_present": true,
    "water_state": "boiling",
    "steam_level": "high",
    "smoke_suspected": false
  },
  "confidence": {
    "water_state": 0.84
  }
}

How to implement quickly:
	•	If you have a working multimodal model locally: ask it to classify into a small label set.
	•	If not: use a small detector/classifier (even a quick transfer-learned MobileNet) for the single event you care about.

2) World model + timer/task state

Aggregate zones + maintain:
	•	active timers
	•	current recipe step index
	•	last alert times (anti-nag)

3) Agent layer (FunctionGemma-style tool calling)

Define tools like:
	•	set_timer(name, seconds)
	•	adjust_timer(name, delta_seconds)
	•	speak(text, priority)
	•	show_card(title, bullets)
	•	mark_step_done(step_id)
	•	reorder_steps(new_order)

Then the LLM outputs JSON tool calls. You execute them locally.

This is where “agentic” becomes undeniable.

How to get the fine-tune done fast (and make it count)

Fine-tune target: “kitchen policy → actions”

You are not trying to teach cooking knowledge from scratch. You’re teaching:
	•	when to interrupt
	•	how to prioritize among multiple concurrent steps
	•	what action to take (timer vs cue vs do nothing)

Training examples look like:
Input:

{
  "stove": {"water_state":"boiling", "pot_present":true},
  "timers": [{"name":"pasta","remaining_s":0}],
  "steps": [{"id":1,"name":"start pasta","status":"pending"}],
  "user_prefs": {"verbosity":"low"}
}

Output:

{
  "actions": [
    {"tool":"speak","args":{"text":"Water’s boiling—add pasta now. Starting 8-minute timer.","priority":"high"}},
    {"tool":"set_timer","args":{"name":"pasta","seconds":480}},
    {"tool":"mark_step_done","args":{"step_id":1}}
  ]
}

Data creation in under an hour
	•	Generate ~150–300 state → action pairs synthetically by enumerating combinations of:
	•	water_state (3–4 values)
	•	0/1/2 active timers
	•	step statuses
	•	severity flags (smoke / boil-over risk)
	•	Use a base model to draft actions, then manually edit the top 50 “core” examples so behavior is crisp.
	•	Fine-tune with LoRA/QLoRA on-device (laptop GPU if available). Even a small run is enough to say “adapted to task.”

What judges like to see:
	•	before/after behavior on the same state snapshot
	•	consistent tool-call formatting after fine-tune
	•	fewer hallucinated actions
